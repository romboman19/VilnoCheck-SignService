# Технічний звіт щодо «файлового КЕП» ПриватБанку для реалізації підпису документів на власному бекенді та фронтенді

## Executive summary

«Файловий КЕП» у контексті ПриватБанку практично означає, що особистий ключ користувача формується/зберігається у **файловому сховищі ключів** (типово **JKS**) і захищається паролем, а відповідальність за конфіденційність і цілісність ключа покладається на клієнта. citeturn17view2turn15view0 Це визначає базовий архітектурний принцип інтеграції: **підпис має виконуватися на стороні клієнта** (браузер/десктоп/мобільний агент) або на сертифікованому пристрої (токен, HSM), а бекенд лише **оркеструє**, **перевіряє**, **зберігає** та **аудитує** підписані артефакти. citeturn17view2turn24view0

Офіційно задокументований для інтеграції з зовнішніми системами “банківський” API підпису у ПриватБанку найбільш явно проявляється не як API “файлового ключа”, а як API **хмарного КЕП SmartID**: `https://acsk.privatbank.ua/cloud/api/back` із ендпоїнтами `/get-certificates`, `/acquire-operation-id`, `/sign`, `/sign-status`, де корисне навантаження проходить через схеми **підпису/шифрування** (CMS/PKCS#7) та повертається як Base64. citeturn37view0turn37view3turn26view0

Для реалізації власного фронту/беку під файловий ключ у більшості українських інтеграцій використовується клас SDK-підхід: криптобібліотеки, що підтримують українські алгоритми та контейнерні формати (зокрема бібліотеки користувача ЦСК від entity["company","АТ «ІІТ»","crypto vendor ukraine"] або open-source бібліотека Cryptonite від PrivatBank IT на entity["company","GitHub","code hosting platform"]). citeturn24view0turn20view0 У наявних офіційних прикладах конфігуруються також мережеві служби інфраструктури довіри ПриватБанку (CMP/TSP/OCSP) у вигляді URL на `http://acsk.privatbank.ua/services/*/`. citeturn37view1turn27view3

Нормативна рамка в Україні задається Законом про електронну ідентифікацію та довірчі послуги, вимогами КМУ, профільними актами entity["organization","Національний банк України","central bank of ukraine"] та вимогами з безпеки entity["organization","Державна служба спеціального зв'язку та захисту інформації України","national cybersecurity regulator ukraine"] (для КНЕДП та їх інфраструктур). citeturn28search0turn28search1turn28search3turn28search2

## Опис сервісу «файловий КЕП» ПриватБанку та екосистема

### Що саме є «файловим КЕП» у ПриватБанку з технічної точки зору

У Регламенті роботи КНЕДП ПриватБанку описано, що відкритий і особистий ключі клієнта можуть генеруватися (а) самостійно на обладнанні клієнта, (б) на робочій станції генерації у надавача/ВПР, або (в) у «хмарному» сховищі ключів надавача. citeturn17view2turn15view0

Ключовий для “файлової” моделі фрагмент: якщо ключ генерується на обладнанні клієнта, то особистий ключ **генерується засобами КЕП у файловому сховищі ключів**, наприклад у форматі **JKS (Java Keystore)**, і **захищається паролем**; відповідальність за конфіденційність і цілісність ключа несе клієнт. citeturn17view2

Практичні індикатори цієї моделі з публічних держсистем: файл КЕП ПриватБанку часто має назву виду `pb_XXXXXXXXXX.jks`, де `XXXXXXXXXX` — податковий номер; назву рекомендують не змінювати (або прибирати автоматичні суфікси на кшталт “(1)”). citeturn31search2turn31search5

### Статус надавача та довірча інфраструктура

ПриватБанк як кваліфікований надавач і його сервіси (зокрема CA/TSA) відображаються у довірчих списках, що публікуються entity["organization","Центральний засвідчувальний орган","ukraine trust list authority"]. На прикладі запису trusted list видно ідентифікатори сервісів, статус “granted” та policy/URI-поля, що посилаються на регламент/інформаційні ресурси надавачів. citeturn12search0

### Відмінність від SmartID як важливий архітектурний вибір

SmartID у ПриватБанку позиціонується як КЕП, що зберігається у захищеному хмарному сховищі, та використовується через мобільний застосунок Приват24 (з погляду UX це зручно для підпису “тут і зараз” без файлів). citeturn31search17turn31search18

Для проєктування вашого рішення це означає:

- **“Файловий КЕП”**: приватний ключ у клієнта (файл/токен), підпис локально, бекенд не повинен отримувати приватний ключ. citeturn17view2turn24view0  
- **SmartID**: підпис у хмарній інфраструктурі ПриватБанку, ваша система викликає API та отримує підпис/статуси по сесії (особливо корисно, якщо ви не хочете/не можете розгортати криптомодулі на клієнті). citeturn37view0turn37view3turn25view2

### Офіційний API хмарного підпису SmartID ПриватБанку

Офіційна інтеграційна пам’ятка описує сервіс SmartID як “сервіс, що надає можливість використання хмарного КЕП” з URL-шляхом `https://acsk.privatbank.ua/cloud/api/back`. citeturn37view0

#### Формати запитів/відповідей та криптообгортки

Документація SmartID описує кілька “об’єктних” рівнів:

- “Зашифрований об’єкт (запит/відповідь)” з полем `encryptedData: String(Base64)`. citeturn26view3  
- “Підписаний об’єкт” із полем `signedData: String(Base64)`. citeturn26view0  
- Окремий формат “Помилки” як JSON із `errorCode`, `errorMessage`, і опційними `additionalErrorCode`, `additionalErrorMessage`. citeturn26view0  
- Важлива умова: “У разі виникнення помилки відповідь не шифрується та не підписується.” citeturn25view2

Практично це означає, що ви повинні реалізувати (або використати надані бібліотеки) наступний ланцюжок: **payload → підпис (CMS/PKCS#7) → шифрування (CMS envelopedData) → транспорт**, а на прийомі — у зворотному порядку, з окремою гілкою для plaintext-ошибок. citeturn26view3turn29search1

#### Авторизація/автентифікація в SmartID API

У прикладах SmartID ідентифікація запитів базується на `clientId` формату `PREFIX_UUID` та `operationId` як ідентифікаторі сесії. citeturn25view3turn37view3 Додатковий рівень довіри досягається криптографічним обгортанням запитів/відповідей та роботою із сертифікатами сервісу підписів (отримуються через `/get-certificates`). citeturn26view3  
**Не описано** (в наявному публічному документі) використання HTTP-заголовків типу OAuth/Bearer або mTLS як обов’язкового механізму доступу; тому цей пункт слід вважати **невизначеним** і уточнювати у ПриватБанку/КНЕДП під час підключення як партнера. citeturn37view0turn26view3

## Вимоги до сертифікатів і форматів: CMS/CAdES/PKCS#7, PKCS#10, X.509, DER/PEM

### Сертифікати, запити на сертифікацію та життєвий цикл

Регламент КНЕДП ПриватБанку фіксує, що відкритий ключ подається на сертифікацію у вигляді **самопідписаного запиту**, а запит на сертифікацію є файлом формату **PKCS#10**. citeturn16view3turn17view2  
Формат PKCS#10 як синтаксис CSR стандартизований в RFC 2986. citeturn36search2

Для X.509-сертифікатів (профіль Internet PKI) базовим референсом є RFC 5280. citeturn29search3  
Перевірка статусу сертифіката “в режимі реального часу” у регламенті описана через OCSP, а актуальний стандарт OCSP описаний в RFC 6960. citeturn16view0turn36search0

Термін дії сертифікатів клієнтів у регламенті ПриватБанку обмежений “не більше двох років”. citeturn16view0turn15view0

## Процес підпису: послідовність кроків для бекенду і фронтенду та сценарії інтеграції

### Референсна модель для “файлового КЕП”: локальний підпис + серверна оркестрація

Нижче — рекомендована модель, яка узгоджується з тим, що особистий ключ є у файловому сховищі (JKS) і за нього відповідає клієнт. citeturn17view2

**Бекенд (ваша система):**
1) Приймає/генерує документ, нормалізує байтове представлення (важливо: підпис має бути над конкретними байтами).  
2) Створює “signing session”: ідентифікатор операції, TTL, anti-replay nonce/challenge, політики підпису (чи потрібен timestamp, чи потрібен підпис+печатка).  
3) Видає фронтенду документ або його хеш (залежить від бібліотеки підпису) і метадані сесії.  
4) Приймає від фронтенду CMS/PKCS#7 (або CAdES) контейнер підпису.  
5) Виконує перевірку: валідність CMS, сертифікатний ланцюжок, статус сертифіката (OCSP/CRL), і за потреби — перевірку позначки часу (TSP). citeturn24view0turn36search0turn29search0turn29search1  
6) Зберігає артефакти (оригінал, підпис, метадані валідації) й формує аудит-трейл.

**Фронтенд (браузер/десктоп/агент):**
1) Дозволяє користувачу вибрати файл ключа `*.jks` і ввести пароль. citeturn31search2turn17view2

### Референсний сценарій SmartID: QR → підтвердження в Приват24 → отримання підпису

SmartID документує, що ваша система формує URL для передачі даних у Приват24 моб. через QR-код. Приклад шаблону: `https://www.privat24.ua/rd/kep?hash=rd/kep/{...json...}`. citeturn37view2turn37view3

- отримує callback на `callbackUrl` (тільки HTTPS:443), або

### Приклади HTTP-запитів і приклади коду (Node.js / Python / JavaScript)

Нижче наведені **прикладні шаблони** (без відтворення повної криптосхеми SmartID, оскільки в реальній інтеграції вона реалізується через надані бібліотеки).

#### Node.js (псевдокод): оркестрація сесії SmartID

```javascript
// Псевдокод: реальна реалізація потребує CMS-підпису та CMS-шифрування за схемою SmartID.

async function smartIdAcquireOperationId({ clientId }) {
  const logicalPayload = { clientId };

  const signed = cmsSign(logicalPayload);      // CMS/PKCS#7 SignedData (DSTU4145)
  const encrypted = cmsEncryptForService(signed); // CMS EnvelopedData шифрування на сертифікат сервісу

  const resp = await fetch("https://acsk.privatbank.ua/cloud/api/back/v1/acquire-operation-id", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ encryptedData: encrypted })
  });

  // Увага: при помилці відповідь може бути plaintext JSON {errorCode, errorMessage}
  const body = await resp.json();

  if (body.errorCode) throw new Error(`${body.errorCode}: ${body.errorMessage}`);

  const signedResp = cmsDecrypt(body.encryptedData);
  const payload = cmsVerifyAndExtract(signedResp);

  return payload.operationId;
}
```

## Обробка помилок, коди помилок, обмеження та продуктивність

### Помилки і статуси у SmartID

SmartID визначає уніфікований формат помилки: `errorCode`, `errorMessage`, плюс опційні додаткові поля. citeturn26view0  
Для `/sign-status` описані статуси:
- `0`: операція зареєстрована  
- `1`: операція розпочата  
- `2`: операція оброблена (підтверджено)  
- `3`: операція відхилена  
- `4`: помилка при обробці запиту клієнтом citeturn25view2  

Результати можуть повертатись як масив `signatures`, де на кожний хеш є або `signature`, або `errorCode/errorMessage` (прикладом у документі фігурує код `504`). citeturn25view2  
Окремо підкреслено: при помилці відповідь може бути не загорнута в криптоконтейнер. citeturn25view2

### Обмеження SmartID, що прямо вказані

| Обмеження | Значення | Джерело |
|---|---:|---|
| Максимальний розмір списку `hashes` | 40 | citeturn25view3turn37view3 |
| Polling `/sign-status`, якщо немає callbackUrl | 1–2 сек протягом 20 сек | citeturn25view2turn25view3 |
| `callbackUrl` | лише HTTPS зі стандартним портом 443 | citeturn25view3turn37view3 |
| Алгоритм/формат | `DSTU4145` + `PKCS7` | citeturn25view3turn37view3 |

**Невизначено в публічній документації SmartID:** ліміти RPS/квоти, максимальні розміри документів (оскільки передаються хеші), політики ретраїв на API-рівні та SLA — потребує узгодження з ПриватБанком під час партнерського підключення. citeturn37view0turn26view3

### Обмеження “файлового” підпису через PrivateSign (як практичний орієнтир)

У публічній інструкції по PrivateSign (не на домені ПриватБанку; тому це **альтернативне/непряме джерело**) вказано, що для підпису потрібні ключі з дійсними сертифікатами зі сховищем `jks` або `dat`, допустимі формати: PDF, Office 97–2003, довільні файли, а розмір файлу має бути 1 KB–10 MB. citeturn34search0turn11search10  
Ці цифри корисні як “інженерна евристика” для UX/перфоманс-тестів, але не як юридично гарантований ліміт сервісів ПриватБанку.

### Орієнтири продуктивності та апаратної безпеки (HSM/токени)

Сторінка опису засобів користувача ЦСК зазначає орієнтири: електронний ключ “Кристал-1” із швидкодією близько 100 мс/ЕЦП, а мережний криптомодуль “Гряда-301” — близько 2000 ЕЦП/с. citeturn24view0  
Ці цифри важливі для планування “батч-підпису”, де підписувачем може бути серверний ключ (HSM) або корпоративний токен у бекенді — але такий дизайн потребує окремої моделі доступу/повноважень і комплаєнсу. citeturn24view0turn28search2

## Безпека, зберігання ключів, TLS, захист від replay/CSRF, логування та аудит

### Зберігання ключів і моделі загроз

Для файлового сценарію регламент прямо фіксує: ключ у файловому сховищі (наприклад JKS) захищається паролем, відповідальність за конфіденційність та цілісність — на клієнті. citeturn17view2  
Отже, архітектурно небезпечним і, як правило, неприйнятним є підхід “завантажити ключ на бекенд і там підписувати” — це різко підвищує ризик компрометації ключа та конфліктує з типовими практиками КЕП/КНЕДП. citeturn17view2turn24view0

Для зменшення ризиків рекомендується підтримка апаратних носіїв/криптомодулів через PKCS#11 та суміжні інтерфейси, що прямо заявлено для бібліотек користувача ЦСК. citeturn24view0

### TLS і транспортна безпека

TLS 1.3 визначений у RFC 8446 як протокол, що запобігає підслуховуванню, підміні та підробці повідомлень. citeturn35search2  
У SmartID окремо зазначено обмеження для callbackUrl: лише HTTPS на 443 порту. citeturn25view3  
Практичний висновок: ваш бекенд має приймати callback лише по HTTPS, з суворою перевіркою host allow-list (щоб уникнути SSRF/помилок маршрутизації), і логувати кореляцію `operationId` ↔ внутрішній документ. citeturn25view3turn35search1

### Захист від replay

У SmartID anti-replay “каркас” реалізується через `operationId` як ідентифікатор сесії та обмежений таймінг опитування/завершення. citeturn25view2turn25view3  
На вашому боці рекомендована додаткова політика:
- TTL на signing-session і одноразовість прийняття результату для конкретного `operationId`. citeturn25view2turn35search1  
- Для timestamp/OCSP протоколів — використання стандартних механізмів (наприклад nonce у TSP за RFC 3161, де це доречно). citeturn29search0

### CSRF для веб-фронтенду

Підпис документів — це state-changing операція, і якщо ваш фронтенд використовує cookie-based session, потрібні анти-CSRF міри. citeturn35search0 Мінімальний набір: CSRF-токени (synchronizer token) або інші рекомендовані OWASP механізми, перевірка Origin/Referer, та заборона змін стану через GET. citeturn35search0turn35search6

### Логування та аудит

OWASP Logging Cheat Sheet акцентує, що без коректного логування та моніторингу неможливо якісно розслідувати інциденти, і дає практичні настанови щодо подій, які варто логувати в security-контексті. citeturn35search1

Для сервісу підпису доцільно формувати аудит-події (з прив’язкою до signing-session), наприклад:
- `SIGN_SESSION_CREATED`, `KEY_LOADED_CLIENT_SIDE` (без витоку секретів), `SIGNATURE_RECEIVED`, `SIGNATURE_VALIDATED`, `CERT_STATUS_CHECKED_OCSP`, `TIMESTAMP_VERIFIED/ADDED`, `SIGN_SESSION_EXPIRED`. citeturn35search1turn36search0turn29search0  

Також доречно враховувати, що нормативні акти України встановлюють вимоги у сфері електронних довірчих послуг та вимоги з безпеки для КНЕДП/їх ВПР; якщо ваш продукт стає частиною регульованого ланцюга, аудит і захист інформації мають узгоджуватися з цими вимогами. citeturn28search0turn28search1turn28search2
