## Варіанти інтеграції з локальним агентом і токеном

### Які механізми доступу до КЕП підтримуються в екосистемі ІІТ

У настанові по web‑бібліотеках підпису описано кілька “провайдерів доступу” з браузера:

- **Агент підпису**: доступ через **JSON‑RPC по HTTP(S) на localhost**. citeturn4view4turn18view2  
- **WebExtension + Native Messaging Host (NMH)**: доступ через stdio‑взаємодію між браузером і локальним модулем. citeturn4view4  
- **NPAPI‑плагін** (застаріле). citeturn4view4  
- **ActiveX компонент** (для legacy сценаріїв). citeturn4view4  

Окремо зазначено, що в середовищах із забороною відкриття фіксованих локальних портів 8081/8083 (наприклад, Azure Virtual Desktop) потрібно використовувати WebExtension + NMH. citeturn20view0

### Як бекенд “взаємодіє” з локальним агентом: реалістична постановка

Важливе уточнення вимог: якщо бекенд розгорнутий **не на ПК користувача**, то він *не може напряму* взаємодіяти з локальним агентом (localhost користувача не маршрутизується до сервера). Тому практично правильний підхід — **фронтенд викликає агент**, а бекенд отримує **вже готовий артефакт підпису**. Це також відповідає ідеї “не передавати приватний ключ/пароль на сервер” (безпековий принцип). citeturn18view2turn25view0

Якщо ж бекенд запускається **локально на тому ж ПК** (наприклад, “desktop‑бекенд” або сервіс у корпоративному контурі), тоді можливі кілька варіантів інтеграції.

## Контракти API бекенд‑сервісу

Нижче — **рекомендований контракт** для вашого “невеликого бекенд‑сервісу”. Він не прив’язаний до конкретного фреймворку та може бути реалізований на Node.js/Python/Java.

### Модель даних

- `FileObject`:
  - `id`, `originalName`, `mimeType`, `sizeBytes`
  - `sha256` (для контролю цілісності на сервері; не плутати з алгоритмами КЕП)
  - `createdAt`, `createdBy` (якщо є акаунти)
- `SigningSession`:
  - `id`, `fileId`
  - `status`: `NEW | READY_FOR_SIGN | SIGNED | VERIFIED | FAILED | EXPIRED`
  - `signatureFormat`: `CMS_DETACHED | CAdES_BASELINE_B | XAdES | PAdES`
  - `clientCert`: опційно — DER/Base64 сертифіката підписувача
  - `signature`: Base64/bytes + метадані
  - `error`: структура помилок (див. нижче)
- `AuditEvent` (опційно): action, timestamp, ip, userAgent, sessionId.

## Підписування і формати підпису

### Кроки зчитування сертифіката/ключа та підпису

Нижче — практичний “енд‑ту‑енд” сценарій, який узгоджується з механікою локального агента та типовими проблемами сертифікатів.

1. **Перевірка наявності агента та портів**
   - UI перевіряє доступність **HTTPS 127.0.0.1:8083** (або HTTP 8081 як fallback). Порти 8081/8083 мають бути прослухувані на 127.0.0.1; у FAQ рекомендують перевірку через `netstat`. citeturn25view0  
   - Якщо агент не працює — показати користувачу інструкцію “запустіть агент” і лінк/QR на інсталятор (офіційні дистрибутиви є у розділі завантажень). citeturn28view0turn13view2  

2. **Забезпечення HTTPS‑довіри до локального агента**
   - Агент генерує SSL‑сертифікат і може імпортувати його у сховище Windows і Firefox; це потрібно, щоб HTTPS‑сайти могли працювати з агентом. citeturn20view3turn25view0  

3. **Білий список довірених сайтів**
   - При першому зверненні вашого домену агент запропонує додати сайт до “Довірених web‑сайтів”. Це критично для безпеки, бо обмежує, які веб‑сторінки можуть використовувати функції підпису. citeturn19view0turn25view0  

4. **Детекція токенів/носіїв**
   - Токен «Кристал‑1» може працювати як CCID‑пристрій; у Windows пристрій повинен коректно відображатися у диспетчері пристроїв (в інструкції наведено приклад перевірки наявності пристрою “ІІТ Е.Ключ Кристал‑1”). citeturn14view2turn17view2  

5. **Ввід пароля**
   - Пароль вводиться на стороні клієнта. Ваша вимога безпеки: пароль **не надсилається на бекенд** і **не логуються** (див. рекомендацію не логувати чутливі дані для WebSocket/подій). citeturn23search3turn25view0

6. **Зчитування сертифіката**
   - На практиці сертифікат підписувача може бути не “всередині токена”, а в каталозі сертифікатів/у ЦСК; помилка “сертифікат не знайдено” прямо пов’язана з відсутністю сертифікатів ЦСК/ЦЗО/користувача або неправильними параметрами. citeturn25view0turn19view0  

7. **Формування підпису**
   - Найпростіший для універсальних файлів варіант: detached CMS/CAdES (вихід: `.p7s` + оригінал).
   - Для PDF за потреби “один файл” — PAdES (вбудовано у PDF). PAdES базується на підписах PDF (ISO 32000‑1) і використовує кодування, еквівалентне CAdES; у профілі PAdES у PDF вбудовується DER‑кодування CMS `SignedData`. citeturn30view1

#### Node.js (Express) — мінімальний бекенд “сесія підпису + підпис”

```js
import express from "express";
import crypto from "crypto";

const app = express();
app.use(express.json({ limit: "2mb" }));

const files = new Map();      // fileId -> { name, bytes, sha256 }
const sessions = new Map();   // sessionId -> { fileId, status, signatureBase64, format }

function sha256(buf) {
  return crypto.createHash("sha256").update(buf).digest("hex");
}

app.post("/api/signing-sessions", (req, res) => {
  const { fileId, signatureFormat, detached } = req.body;
  if (!files.has(fileId)) return res.status(404).json({ error: "FILE_NOT_FOUND" });

  const sessionId = "ssn_" + crypto.randomBytes(12).toString("hex");
  sessions.set(sessionId, { fileId, signatureFormat, detached, status: "READY_FOR_SIGN" });

  res.json({
    id: sessionId,
    status: "READY_FOR_SIGN",
    uploadSignatureUrl: `/api/signing-sessions/${sessionId}/signature`
  });
});

app.post("/api/signing-sessions/:id/signature", (req, res) => {
  const s = sessions.get(req.params.id);
  if (!s) return res.status(404).json({ error: "SESSION_NOT_FOUND" });

  const { signatureBase64, signatureFormat, fileSha256 } = req.body;
  const f = files.get(s.fileId);

  // Мінімальний контроль: підпис прив’язується до конкретного файлу
  if (fileSha256 && fileSha256 !== f.sha256) {
    s.status = "FAILED";
    return res.status(409).json({ error: "FILE_HASH_MISMATCH" });
  }

  s.signatureBase64 = signatureBase64;
  s.signatureFormat = signatureFormat;
  s.status = "SIGNED";

  res.json({
    status: "SIGNED",
    downloadUrl: `/api/signing-sessions/${req.params.id}/download`
  });
});
```

> Примітка: реальна реалізація потребує upload файлів (`/api/files`) та формування пакету видачі (наприклад, `zip` з оригіналом і `.p7s`). Також не забувайте про CSRF‑захист state‑changing endpoint’ів (OWASP). citeturn23search0

#### Python (FastAPI) — приклад структури endpoint’ів

```py
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()
files = {}
sessions = {}

class CreateSessionReq(BaseModel):
    fileId: str
    signatureFormat: str
    detached: bool = True

class UploadSignatureReq(BaseModel):
    signatureBase64: str
    signatureFormat: str
    fileSha256: str | None = None

@app.post("/api/signing-sessions")
def create_session(req: CreateSessionReq):
    if req.fileId not in files:
        raise HTTPException(status_code=404, detail="FILE_NOT_FOUND")
    sid = "ssn_" + "TODO_RANDOM"
    sessions[sid] = {"fileId": req.fileId, "status": "READY_FOR_SIGN", "format": req.signatureFormat}
    return {"id": sid, "status": "READY_FOR_SIGN", "uploadSignatureUrl": f"/api/signing-sessions/{sid}/signature"}

@app.post("/api/signing-sessions/{sid}/signature")
def upload_signature(sid: str, req: UploadSignatureReq):
    if sid not in sessions:
        raise HTTPException(status_code=404, detail="SESSION_NOT_FOUND")
    sessions[sid]["status"] = "SIGNED"
    sessions[sid]["signatureBase64"] = req.signatureBase64
    return {"status": "SIGNED", "downloadUrl": f"/api/signing-sessions/{sid}/download"}
```

## Безпека та UX

### Безпекові вимоги

1. **Пароль токена**
   - Пароль вводиться у браузері і *передається тільки локальному агенту*; бекенд пароля не отримує, не зберігає і не логатує. Це узгоджується з практиками логування (не логувати токени/секрети) та з моделлю “підпис на клієнті”. citeturn23search3turn25view0  

2. **Захист від “drive‑by” доступу до localhost**
   - Локальний агент підтримує список **довірених сайтів**; ваш UX має чітко інструктувати користувача додати **лише ваш production‑домен** (і, за потреби, ваш staging‑домен). citeturn19view0turn25view0  

3. **TLS**
   - Бекенд обов’язково працює через HTTPS; TLS 1.3 прямо декларує дизайн проти підслуховування, підміни та підробки повідомлень. citeturn23search2  
   - Для взаємодії з агентом бажано використовувати HTTPS (8083), оскільки в екосистемі описано наявність HTTPS‑порту і кореневого сертифіката агента в сховищах. citeturn25view0turn20view3  

4. **CORS**
   - CORS є механізмом контрольованих крос‑доменних запитів у браузері; на бекенді потрібен allow‑list ваших origin’ів, без wildcard для чутливих endpoint’ів. citeturn23search1turn23search5  

5. **CSRF**
   - Для всіх state‑changing endpoint’ів (`POST/PUT/DELETE`) застосувати CSRF‑мітигації: CSRF‑токени (чи вбудовані механізми фреймворку) і перевірка на бекенді. citeturn23search0  

6. **Обмеження доступу до локального агента**
   - Мінімально: покладатися на allow‑list “Довірені web‑сайти” агента. citeturn19view0turn25view0  
   - Додатково (рекомендація архітектури): робити “handshake” у вашому фронтенді з бекенд‑nonce (одноразовий виклик, який підписується локально), щоб бекенд приймав підпис лише для активної сесії (зменшує ризик replay у вашому домені).

7. **Логування**
   - Локальний агент може записувати події в системний журнал і відправляти інформацію агенту моніторингу; це корисно для support‑сценаріїв, але потрібно узгодити політики приватності. citeturn24view1  
   - На бекенді: не логувати тіло підпису/сертифікати повністю в INFO‑логах; зберігати лише хеші/короткі метадані.
