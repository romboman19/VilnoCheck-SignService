# Робота КЕП в Україні та підпис фіскальних чеків: технічний і процедурний аналіз з фокусом на практики ПриватБанку

## Executive summary

Кваліфікований електронний підпис (КЕП) в Україні формально визначений як удосконалений електронний підпис, який **створюється із застосуванням засобу КЕП** і **базується на кваліфікованому сертифікаті**; окремо закон визначає удосконалений підпис на кваліфікованому сертифікаті, який **не містить ознаки зберігання особистого ключа в засобі КЕП**.

Для практичної фіскалізації чеків у контексті ПРРО ключове: ДПС надає **API фіскального сервера** (продакшн і тестове), де **чек передається як XML**, а **підпис — окремим полем `check_sign` (bytes)**; після відкриття зміни всі чеки до її закриття мають бути підписані тим самим підписом — це впливає на UX (кешування пароля/сесії) і безпеку.

У практиках ПриватБанку чітко простежуються два «світи»:
- **SmartID** як хмарний КЕП (підтвердження в мобільному застосунку, пароль/біометрія) — активно використовується в їхньому ПРРО «Каса».
- **файлові ключі** (контейнери на кшталт `.p12/.pfx`) — у бізнес-процесах зустрічаються як «файловий підпис» і застосовуються для реєстраційних/облікових сценаріїв, але сам ПриватБанк у документації ПРРО «Каса» позиціонує файловий підпис як УЕП із нижчим рівнем довіри, тоді як SmartID — як КЕП.

Технічно процес «підписати фіскальний чек» майже завжди зводиться до: **підготовка XML → нормалізація/серіалізація → обчислення хешу → створення підпису (часто у CMS/CAdES контейнері) → передача чека й підпису в ДПС → валідація (ланцюжок сертифікатів, статус, позначка часу/OCSP/CRL за політикою)**.

## Нормативна рамка та вимоги до підпису в е-документообігу і фіскалізації

Законодавча «вісь» для КЕП/довірчих послуг — закон про електронну ідентифікацію та електронні довірчі послуги. Він містить визначення КЕП, кваліфікованого надавача, кваліфікованих сертифікатів, а також перелік умов підтвердження чинності КЕП.

Після змін, пов'язаних з імплементацією підходів ЄС, уряд окремо затвердив вимоги до **форматів удосконалених підписів/печаток** для електронних публічних послуг. Постанова дозволяє використання форматів на базових рівнях **B-B/B-T/B-LT/B-LTA** та прямо перераховує представлення: **XML, CMS, PDF, ASN.1, JSON** або контейнер підписів — за умови відповідності стандартам із додатка.

Для «фіскального чеку» важливий не лише підпис, а й **структура/форма електронного розрахункового документа**. Вона регулюється наказом Мінфіну щодо форми і змісту розрахункових документів/електронних розрахункових документів.

На технічному рівні ДПС публікує опис API для взаємодії фіскального сервера та ПРРО, визначаючи продакшн/резервну адресу та тестовий контур для розробників.

## Технічні стандарти КЕП і формати підписів

### Порівняння форматів підпису

| Формат | Базовий стандарт | Типова «форма» в системах | Сильні сторони | Типові ризики/пастки в розробці |
|---|---|---|---|---|
| XMLDSig | XMLDSig | `<Signature>` в XML або detached | Дуже точний контроль над тим, що саме підписується (XPath/Transforms) | Канонікалізація (C14N), пробіли/неймспейси, різні серіалізації дають різні хеші |
| XAdES | XAdES (ETSI) | XMLDSig + qualifying properties | «Підпис із доказами» (час, політики) поверх XML | Складні профілі, сумісність валідаторів, вимоги до time-stamp/OCSP |
| CMS/PKCS#7 | CMS (RFC 5652) | ASN.1 контейнер (DER), часто Base64 | Простий detached-підпис довільних байтів | Узгодити, що є «саме даними» (encoding), що інкапсульовано, які атрибути підписані |
| CAdES | ETSI CAdES + CMS | CMS SignedData з атрибутами CAdES | Підтримка довготривалої чинності (LT/LTA) | Правильна політика валідації, вбудовування/перевірка доказів статусу сертифіката |
| ASiC | ETSI ASiC (ZIP) | `.asice/.asics` контейнер | Зручно для «файл + підпис + метадані» | Потрібна сувора структура ZIP/маніфестів, сумісність профілів |

## Носії ключів, сертифікати та криптографічний стек

### Контейнер ключа і «пароль ключа»

Найпоширеніший «файловий ключ» — контейнер **PKCS#12** (`.p12/.pfx`). Стандарт описує перенесення «особистої ідентичності» користувача: приватні ключі, сертифікати, додаткові секрети/об'єкти; контейнер може бути зашифрований і захищений паролем.

В українських довідкових ресурсах «файловий носій» описується як файл із особистим ключем (наприклад, `Key-6.*`, включно з `.pfx/.p12`), який потрібно вибрати і **ввести пароль доступу до особистого ключа**.

### Токени, PIN і PKCS#11

Апаратні токени/смарткарти/НSM працюють через **PKCS#11 (Cryptoki)**, який визначає API для доступу до криптографічних токенів.

Ключова різниця з `.p12/.pfx`: PIN на токені зазвичай **не розшифровує ключ у вашому процесі** — він відкриває доступ до об'єкта ключа всередині пристрою, а підпис виконується в межах токена.

### КЕП vs УЕП і «ознака захищеного носія»

Навіть якщо зробити копію ключа з файлового носія на захищений носій, це **не означає**, що ключ «згенерований на захищеному носії» (і відповідна ознака в сертифікаті не з'явиться ретроспективно).

### Криптоалгоритми та зміни 2026 року

Перехід на стандарт «Купина»: нові КЕП формуються з використанням сучаснішого алгоритму, старі сертифікати не втрачають чинності одразу.

## Процес підпису фіскального чеку для ПРРО

### Транспорт і API ДПС

У протоколі фіскалізації:
- чек передається як XML (`check`),
- підпис передається окремо (`check_sign`) як bytes,
- після відкриття зміни всі чеки до її закриття мають бути підписані одним підписом.

### Канонічний пайплайн підпису

1. Підготовка даних чеку
2. Формування XML за схемою ДПС/Мінфіну
3. Детермінована серіалізація: UTF-8, стабільні поля, порядок
4. Формат підпису (CMS/CAdES detached або XMLDSig/XAdES)
5. Хеш/атрибути SignedData
6. Підпис приватним ключем
7. Формування контейнера підпису + сертифікат(и)
8. Валідація локально: cert chain + OCSP/CRL + час
9. Відправка до ДПС: check + check_sign
10. Квитанція/відповідь: фіскальний номер/помилки

### Практики ПриватБанку

- SmartID як хмарний КЕП у ПРРО «Каса»
- SmartID API: `/cloud/api/back/v1/...` — сертифікати, сесія, підпис
- Файлові ключі — УЕП із нижчим рівнем довіри

### Бібліотеки

| Мова/платформа | Читання PKCS#12 | CMS/CAdES підпис | Коментарі |
|---|---|---|---|
| PHP | `openssl_pkcs12_read` | `openssl_pkcs7_*` | Для українських алгоритмів може не вистачити stock OpenSSL |
| Python | `pkcs12.load_key_and_certificates` | потребує додаткових модулів | `cryptography` документує PKCS#12 як RFC 7292 |
| Java | `KeyStore PKCS12` + Bouncy Castle | `CMSSignedDataGenerator` | Bouncy Castle — де-факто стандарт |
| Browser/JS | Залежно від SDK | JS-бібліотека від провайдера | Потрібні криптографічні бібліотеки від провайдера/агента |

## Висновки

- КЕП = носій ключа + сертифікат + формат підпису + політика перевірки + процедурні ролі + правила ДПС
- Для ПРРО: чек і підпис передаються окремо (`check` і `check_sign`), підпис консистентний в межах зміни
- Ризик 2026: перехід на «Купину» — нові OID/алгоритми
